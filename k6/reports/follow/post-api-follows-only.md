# POST /api/follows 부하 테스트 결과

## 테스트 목적

`POST /api/follows` API가 신규 팔로우 생성 요청을 안정적으로 처리할 수 있는지 확인한다.

이번 테스트는 운영 시나리오 전체가 아니라 팔로우 생성 API 단독 테스트다. 따라서 본 부하 구간에서는 프로필 조회, 팔로우 여부 조회, 로그인 요청을 제외하고 `POST /api/follows`의 신규 생성 처리량과 응답 시간을 중심으로 판단한다.

확인하려는 핵심 지표는 다음과 같다.

- 신규 팔로우 생성 성공 수
- 목표 RPS 대비 실제 처리량
- 요청 실패율
- p95, p99 응답 시간
- 최대 응답 시간
- 중복 팔로우 발생 여부

## 테스트 조건

테스트 결과 파일은 [post-api-follows-only.html](./post-api-follows-only.html)이다.

테스트 스크립트는 [follow-create.js](../../follow-create.js)를 사용했다.

부하 방식은 `ramping-arrival-rate`다. VU 수를 직접 목표로 잡는 대신, 초당 요청 수를 단계적으로 증가시키는 방식이다.

```text
10 RPS 1분
30 RPS 1분
50 RPS 1분
100 RPS 1분
150 RPS 1분
200 RPS 1분
0 RPS 30초
```

테스트 데이터 조건:

```text
FOLLOWER_COUNT=1000
FOLLOWEE_COUNT=3000
최대 고유 팔로우 조합 수=3,000,000
```

요청 조합은 같은 실행 안에서 중복되지 않도록 `(follower, followee)` 조합을 순차적으로 선택한다. 따라서 `409 Conflict`가 발생하면 테스트 데이터가 남아 있거나 시나리오가 잘못된 것으로 해석한다.

## 조건 선정 이유

`POST /api/follows` 단독 성능을 확인할 때는 DAU, MAU보다 피크 시간대의 초당 요청 수가 더 직접적인 기준이다.

팔로우 생성은 쓰기 요청이므로 다음 항목이 중요하다.

- 초당 신규 INSERT 요청을 어느 정도까지 처리하는지
- 부하 증가 시 p95, p99가 언제부터 증가하는지
- 실패율이 1% 미만으로 유지되는지
- DB connection, lock, unique constraint 경합이 발생하는지

따라서 `ramping-vus`보다 `ramping-arrival-rate`가 더 적절하다. `ramping-arrival-rate`는 목표 RPS를 직접 지정할 수 있어, 개선 전후 동일한 요청률에서 성능을 비교하기 쉽다.

## 테스트 결과

전체 HTTP 지표:

```text
전체 요청 수: 33,699건
초당 요청 수: 47.59
전체 실패율: 0.00%
체크 성공률: 100.00%
평균 응답 시간: 29.12 ms
중앙값: 13.61 ms
p90: 74.33 ms
p95: 81.33 ms
p99: 173.31 ms
최대 응답 시간: 710.91 ms
```

`POST /api/follows` 지표:

```text
요청 수: 29,699건
신규 팔로우 생성 성공 수: 29,699건
중복 팔로우 수: 0건
요청 실패율: 0.00%
TPS: 41.94
평균 응답 시간: 22.37 ms
중앙값: 12.67 ms
p90: 37.21 ms
p95: 62.31 ms
p99: 178.22 ms
최대 응답 시간: 710.91 ms
```

## 결과 분석

이번 테스트에서는 중복 팔로우가 발생하지 않았다.

```text
중복 팔로우 수: 0건
신규 팔로우 생성 성공 수: 29,699건
```

따라서 이번 결과는 이전처럼 `409 Conflict`가 섞인 요청 처리 결과가 아니라, 신규 팔로우 생성 중심의 결과로 볼 수 있다.

응답 시간도 양호하다.

```text
POST /api/follows p95: 62.31 ms
POST /api/follows p99: 178.22 ms
POST /api/follows max: 710.91 ms
```

p95가 100ms 미만이고, p99도 200ms 미만이다. 최대 응답 시간이 710.91ms까지 튄 요청은 있었지만, 전체 실패율은 0.00%이고 체크 성공률도 100.00%였다.

따라서 현재 테스트 조건에서는 `POST /api/follows`가 신규 팔로우 생성 요청을 안정적으로 처리했다고 볼 수 있다.

## TPS 해석 주의

리포트의 TPS는 41.94로 표시되었다.

이 값은 “200 RPS를 유지했을 때의 TPS”가 아니라, 전체 테스트 실행 시간 기준으로 계산된 평균이다. 특히 k6 summary의 custom counter rate는 setup 시간의 영향을 받을 수 있다.

현재 스크립트는 setup 단계에서 follower와 followee 로그인 정보를 준비한다.

```text
follower 1000명 로그인
followee 3000명 로그인
```

이 setup 시간이 전체 rate 계산에 포함되면, 본 부하 구간에서 더 높은 RPS를 처리했더라도 평균 TPS는 낮게 보일 수 있다.

따라서 이번 결과에서 확실히 말할 수 있는 것은 다음이다.

```text
29,699건의 신규 팔로우 생성 요청을 실패 없이 처리했다.
POST /api/follows의 p95는 62.31ms였다.
중복 팔로우는 발생하지 않았다.
```

반면 다음 결론은 아직 별도 테스트가 필요하다.

```text
POST /api/follows가 200 RPS 신규 INSERT 부하를 5분 이상 안정적으로 처리한다.
```

## 다음 테스트 제안

200 RPS 안정성을 명확히 검증하려면 ramp 탐색이 아니라 200 RPS hold 테스트가 필요하다.
