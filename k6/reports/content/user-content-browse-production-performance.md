# 운영환경 콘텐츠 탐색 시나리오 성능 테스트 보고서

## 1. 테스트 요약

현재 `moduply.co.kr` 운영환경의 애플리케이션 인스턴스 **1대**를 대상으로 콘텐츠 탐색 시나리오를 검증했다.

목표 부하 테스트는 사용자 여정 50 TPS, 전체 약 150 RPS를 목표로 실행했다. 이 테스트에서 HTTP 실패율은 0%였고 전체 응답시간 p95는 64.77ms였다.

다만 저장된 결과 파일에는 `dropped_iterations` count와 `count==0` threshold 판정이 포함되어 있지 않다. 따라서 목표 도착률만큼 사용자 여정이 실행됐는지 확인할 수 없으므로, 전체 약 150 RPS를 검증된 안정 처리량으로 확정하지 않고 **HTTP 응답 지표 기준 잠정 결과**로 해석한다. 추가 한계 탐색에서는 전체 요청이 순간 약 312 RPS에 이르는 구간까지 HTTP 오류는 없었지만, 응답시간이 크게 증가하고 필요한 VU가 최대치에 도달해 테스트를 조기 중단했다.

## 2. 이 시나리오를 선택한 이유

테스트는 실제 사용자가 서비스를 탐색할 때 반복적으로 발생하는 다음 흐름을 재현한다.

```text
로그인 → 콘텐츠 목록 조회 → 콘텐츠 상세 조회 → 리뷰 목록 조회
```

이 흐름을 선택한 이유는 다음과 같다.

- 콘텐츠 탐색은 서비스의 대표적인 읽기 중심 사용자 여정이다.
- 목록, 상세, 리뷰 조회를 연속 호출해 단일 API가 아닌 실제 요청 조합의 처리 성능을 확인할 수 있다.
- 콘텐츠 목록 캐시, 콘텐츠·리뷰 DB 조회, 사용자 인증, DTO 변환 및 이미지 URL 생성처럼 주요 읽기 경로가 함께 사용된다.
- 로그인한 여러 사용자의 JWT를 순환 사용해 단일 토큰만 반복 호출하는 비현실적인 테스트를 피했다.
- 인기 콘텐츠에 요청이 집중되는 운영 특성을 반영해 캐시 효과와 특정 데이터 집중 상황을 함께 확인했다.

로그인은 테스트 시작 단계에서 계정 10개로 한 번씩만 수행했다. 따라서 본 테스트의 처리량은 로그인 API의 반복 성능이 아니라, 로그인된 사용자의 콘텐츠 탐색 성능을 측정한다.

## 3. 공통 테스트 조건

| 항목 | 조건 |
|---|---|
| 대상 | `https://moduply.co.kr` |
| 운영 애플리케이션 인스턴스 | 1대 |
| 부하 도구 | k6, Docker Compose |
| 실행 모델 | `ramping-arrival-rate` |
| 테스트 사용자 | 운영 부하 테스트용 계정 10개 |
| 사용자 선택 | 테스트 전체 iteration에서 10개 JWT 순환 사용 |
| 콘텐츠 목록 | 20개, 평점 내림차순 |
| 리뷰 목록 | 20개, 생성일 내림차순 |
| 인기 데이터 집중 | 상위 10개 콘텐츠에 80% 확률로 접근 |
| 사용자 대기시간 | API 사이 0.2~0.8초 |
| 성공 기준 | HTTP 실패율 1% 미만, 체크 성공률 99% 초과, `dropped_iterations` count 0건 |

`ramping-arrival-rate`를 사용한 이유는 서버 응답이 느려져도 VU 반복 횟수에 처리량이 종속되지 않도록 하고, 목표 요청 도착률을 기준으로 서비스가 부하를 감당하는지 확인하기 위해서다.

한 번의 사용자 여정은 조회 API 3개를 호출한다. 따라서 사용자 여정 50 TPS는 API별 약 50 TPS이며, 서비스 전체로는 약 150 RPS에 해당한다.

> 현재 보관된 HTML 결과에는 `dropped_iterations` 지표와 threshold 결과가 없다. 아래 표의 `확인 불가`는 0건을 의미하지 않으며, 해당 기준의 통과 여부를 판정할 수 없다는 의미다.

## 4. 목표 부하 테스트 (안정성 판정 보류)

### 실행 조건

| 항목 | 값 |
|---|---:|
| 시작 사용자 여정 | 20 TPS |
| 단계 | 20 → 35 → 50 → 50 TPS |
| 단계별 시간 | 2분 |
| 종료 감속 | 30초 |
| 사전 할당 VU | 80 |
| 최대 VU | 150 |

50 TPS 구간을 한 단계 더 유지해 단순히 목표치에 도달하는 것뿐 아니라 해당 부하를 지속해서 처리하는지도 확인했다.

### 전체 결과

| 지표 | 결과 |
|---|---:|
| 완료한 사용자 여정 | 17,537회 |
| 전체 요청 | 52,621건 |
| 테스트 전체 평균 요청률 | 102.92 RPS |
| 목표 유지 구간 요청률 | 약 150 RPS |
| HTTP 실패율 | **0.00%** |
| 체크 성공률 | **100.00%** |
| `dropped_iterations` | **확인 불가** — 기존 결과 파일에 미포함 |
| `dropped_iterations` threshold (`count==0`) | **판정 불가** |
| 평균 응답시간 | 35.38ms |
| p90 | 54.95ms |
| p95 | **64.77ms** |
| p99 | 207.21ms |
| 최대 응답시간 | 1,172.98ms |
| 최종 판정 | **보류** — 목표 도착률 달성 여부 확인 필요 |

전체 평균 RPS가 150보다 낮은 이유는 20 TPS부터 단계적으로 증가한 구간과 종료 감속 구간이 평균에 포함되기 때문이다.
그러나 `dropped_iterations`가 보존되지 않아 목표 유지 구간에서 50 TPS가 누락 없이 실행됐는지는 확인할 수 없다.

| API | 요청 수 | 실패율 | 평균 | p95 | p99 | 최대 |
|---|---:|---:|---:|---:|---:|---:|
| 콘텐츠 목록 `GET /api/contents` | 17,537 | 0.00% | 62.98ms | **100.11ms** | 408.52ms | 1,172.98ms |

단일 운영 인스턴스를 대상으로 전체 약 150 RPS를 목표로 실행한 결과 HTTP 오류 없이 낮은 p95 응답시간을 기록했다. 세 조회 API 중 콘텐츠 목록이 가장 무거웠지만 p95 100.11ms로 양호했고, 콘텐츠 상세와 리뷰 목록은 p95 35ms 이내였다. 다만 `dropped_iterations` count를 확인하기 전에는 이 결과를 150 RPS 안정 처리 성능으로 확정할 수 없다.

| 계획한 최대 전체 요청률 | 약 500 RPS |

## 6. 결론

현재 단일 운영 인스턴스의 측정 결과와 판정 한계는 다음과 같이 정리할 수 있다.

- 실제 콘텐츠 탐색 사용자 흐름에서 **사용자 여정 50 TPS**, 전체 약 150 RPS를 목표로 테스트했다.
- 완료한 사용자 여정은 17,537회이고 전체 요청은 52,621건이었다.
- 목표 부하 테스트에서 HTTP 실패율 0%, 체크 성공률 100%, 전체 p95 64.77ms를 기록했다.
- API별 50 TPS에서 콘텐츠 목록 p95 100.11ms, 상세 p95 34.83ms, 리뷰 p95 33.81ms였다.
- 저장된 결과에 `dropped_iterations` count와 threshold 판정이 없어 목표 도착률 달성 여부를 확인할 수 없다.
- 따라서 **전체 약 150 RPS 안정 처리** 판정은 보류하며, `dropped_iterations==0`을 확인한 재실행 결과가 필요하다.
- 약 312 RPS까지 요청 오류는 없었지만 p95가 1.95초로 증가했으므로 이 수치는 안정 처리량이 아니라 포화 탐색 결과다.

현재 결과로 설명할 수 있는 보수적인 표현은 다음과 같다.

> 모두의 플리 운영환경의 단일 애플리케이션 인스턴스를 대상으로 전체 약 150 RPS를 목표로 콘텐츠 탐색 시나리오를 실행했으며, 완료된 요청의 HTTP 실패율은 0%, 전체 응답시간 p95는 64.77ms였다. 다만 `dropped_iterations` 실적이 보존되지 않아 목표 도착률을 누락 없이 처리했는지는 확인되지 않았다.

이 결과는 해당 조회 시나리오와 테스트 데이터 분포에 한정된다. `dropped_iterations` 검증 전에는 안정 처리량을 확정할 수 없으며, 서비스 전체 API, 쓰기 요청, 장시간 운영 안정성, 순간 트래픽 및 장애 상황의 성능을 대표하지도 않는다.

## 7. 후속 검증 권장사항

- 150~300 RPS 사이를 더 작은 단계로 나눠 단일 인스턴스의 정확한 포화 시작점을 측정한다.
- 고부하 구간에서 애플리케이션 CPU·메모리·GC·스레드·DB 커넥션 풀·Redis·RDB 지표를 함께 수집한다.
- Cold Cache와 Warm Cache를 분리해 콘텐츠 목록 성능 차이를 확인한다.
- 재실행 시 k6 summary 원문 또는 JSON 결과를 보존하고 `dropped_iterations` count와 `count==0` threshold 판정을 각 결과 표에 기록한다.
- `dropped_iterations==0`을 확인한 뒤 전체 약 150 RPS 안정 처리 여부를 최종 판정한다.
- 인스턴스를 확장한 뒤 동일 부하를 재실행해 처리량 증가와 공용 DB·Redis 병목 여부를 비교한다.
