# 콘텐츠 목록 조회 성능 개선

## 문제

콘텐츠 기본 목록 조회 API(`/api/contents`)는 부하테스트에서 응답 시간이 길고 처리량이 낮게 측정됐다.

개선 전 측정 결과는 다음과 같다.

| 항목 | 값 |
|---|---:|
| 총 요청 수 | 17,331 |
| 초당 요청 수 | 49.08 |
| 평균 응답 시간 | 8,349.99 ms |
| p95 응답 시간 | 21,133.47 ms |
| p99 응답 시간 | 22,973.28 ms |
| 최대 응답 시간 | 24,577.21 ms |
| 성공 트랜잭션 수 | 17,331 |
| 요청 실패율 | 0.00% |

주요 원인은 목록 조회 요청마다 콘텐츠, 태그, 평점, 리뷰 수, 현재 시청자 수, 썸네일 URL을 조합하면서 DB와 Redis 조회가 반복되는 구조였다.

Redis 캐시 적용 초기에는 캐시 값 역직렬화 시 `LinkedHashMap`으로 복원되어 `ContentListCachePageDto`로 캐스팅하지 못하는 문제가 발생했다. 또한 부하 상황에서 Redis 연결을 안정적으로 확보하지 못해 `RedisConnectionFailureException`이 발생했다.

## 해결과정

콘텐츠 목록 조회 결과를 Redis `contentList` 캐시에 저장하도록 변경했다. 최신순 목록 조회(`sortBy=createdAt`)는 목록 전용 캐시 서비스를 통해 조회하며, 캐시 키에는 조회 조건을 포함한다.

```text
moduply:cache:v3:contentList::null,null,,null,null,20,createdAt,DESCENDING
```

Redis 값 역직렬화 문제는 `GenericJackson2JsonRedisSerializer`와 타입 정보를 포함한 `ObjectMapper` 설정으로 해결했다. 기존 캐시 구조와 충돌하지 않도록 cache prefix도 `v3`로 변경했다.

목록 캐시에는 다음 값을 포함했다.

- 콘텐츠 ID
- 콘텐츠 타입
- 제목
- 설명
- 썸네일 URL
- 태그
- 평균 평점
- 리뷰 수

현재 시청자 수는 실시간성이 더 중요하므로 캐시에 포함하지 않고, 조회 시점에 `WatchingSession` 기준으로 계산해 최종 응답에 조합한다. 여러 콘텐츠의 시청자 수 조회는 Redis pipeline으로 처리하도록 개선했다.

평점과 리뷰 수를 캐시에 포함하면서 생길 수 있는 최신성 문제는 리뷰 통계 갱신 시점에 `contentList` 캐시를 evict하는 방식으로 해결했다. 리뷰 생성, 수정, 삭제 후 `refreshReviewStats()`가 호출되면 목록 캐시가 삭제되고 다음 조회에서 최신 값으로 다시 캐싱된다.

부하테스트 중 Redis 연결 실패를 줄이기 위해 Lettuce connection pool을 활성화하고 `commons-pool2` 의존성을 추가했다. 또한 `/api/contents` 요청마다 반복 출력되던 INFO 로그를 DEBUG로 낮추고, 부하테스트 실행 시 SQL 로그를 비활성화했다.

`contentList` TTL은 30초에서 5분으로 늘렸다. 콘텐츠 변경과 리뷰 통계 변경 시 명시적으로 evict하므로, 테스트 중 불필요한 캐시 재생성 비용을 줄이기 위한 조정이다.

## 결과

최종 개선 후 측정 결과는 다음과 같다.

| 항목 | 개선 전 | 개선 후 |
|---|---:|---:|
| 총 요청 수 | 17,331 | 29,852 |
| 초당 요청 수 | 49.08 | 81.88 |
| 평균 응답 시간 | 8,349.99 ms | 442.88 ms |
| p95 응답 시간 | 21,133.47 ms | 2,801.25 ms |
| p99 응답 시간 | 22,973.28 ms | 7,231.81 ms |
| 최대 응답 시간 | 24,577.21 ms | 8,613.55 ms |
| 성공 트랜잭션 수 | 17,331 | 29,848 |
| 요청 실패율 | 0.00% | 0.01% |

개선 효과는 다음과 같다.

| 지표 | 개선 효과 |
|---|---:|
| 평균 응답 시간 | 약 94.7% 감소 |
| p95 응답 시간 | 약 86.7% 감소 |
| p99 응답 시간 | 약 68.5% 감소 |
| 초당 요청 수 | 약 66.8% 증가 |
| 총 처리 요청 수 | 약 72.2% 증가 |

최종적으로 콘텐츠 기본 목록 조회는 매 요청마다 DB에서 목록 정보를 조립하는 구조에서 Redis 캐시 기반 조회 구조로 변경되었다. 평점과 리뷰 수는 캐시에 포함하고, 현재 시청자 수만 조회 시점에 Redis에서 계산해 조합하는 방식으로 개선했다.
