# 리뷰 목록 조회 부하 테스트 및 성능 개선기

## 1. 테스트 목적

리뷰 목록 조회 API는 인기 콘텐츠에 트래픽이 몰릴 때 동일한 첫 페이지 요청이 반복되는 특성이 있다.  
기존에는 DB 인덱스를 적용했음에도 고부하 상황에서 응답 시간이 크게 튀는 구간이 있었기 때문에, Redis 캐시 적용 전후의 성능을 비교했다.

비교 대상 리포트는 다음과 같다.

| 구분 | 리포트 |
|---|---|
| 캐시 적용 전 | `review-list-index.html` |
| 캐시 적용 후 | `review-list-cache.html` |

## 2. 테스트 조건

두 리포트 모두 `review-list-default.js` 기반의 리뷰 목록 조회 부하 테스트 결과다.

테스트 시나리오는 인기 콘텐츠에 요청이 집중되는 `hot` 모드 기준으로 해석했다.

```powershell
docker compose -f docker-compose-k6.yml run --rm --service-ports `
  -e BASE_URL=http://host.docker.internal:8080 `
  -e USER_COUNT=1000 `
  -e CONTENT_POOL_SIZE=10 `
  -e REVIEW_LIST_MODE=hot `
  -e HOT_CONTENT_COUNT=10 `
  -e K6_WEB_DASHBOARD_EXPORT=/scripts/reports/review/review-list-cache.html `
  k6 run /scripts/review-list-default.js
```

주의할 점은 HTML 리포트 자체에는 모든 환경 변수가 저장되어 있지 않다는 것이다. 따라서 테스트 조건은 실행 명령과 리포트 파일명을 기준으로 정리했다.

## 3. 캐시 적용 방식

기존 흐름은 다음과 같았다.

```text
리뷰 목록 요청
-> ReviewService
-> ReviewQDSLRepository
-> PostgreSQL 조회
-> 작성자/프로필 이미지 조회
-> 응답 생성
```

개선 후 흐름은 다음과 같다.

```text
리뷰 목록 첫 페이지 요청
-> ReviewService
-> ReviewListCacheService
-> Redis 캐시 조회
   -> 캐시 hit: DB 조회 없이 응답
   -> 캐시 miss: DB 조회 후 Redis 저장
```

적용 범위는 첫 페이지 조회로 제한했다.

- 캐시 이름: `reviewList`
- TTL: 30초
- 캐시 key: `contentId`, `limit`, `sortBy`, `sortDirection`
- 캐시 조건: `cursor == null && idAfter == null`
- 리뷰 생성/수정/삭제 시 `reviewList` 전체 캐시 무효화

첫 페이지로 제한한 이유는 인기 콘텐츠의 최초 진입 요청이 가장 반복성이 높고, 커서 기반 다음 페이지는 사용자별 탐색 위치가 달라 캐시 효율이 낮기 때문이다.

## 4. 결과 비교

아래 표는 리포트의 “리뷰 목록조회 API” 섹션 기준이다.

| 지표 | 캐시 적용 전 | 캐시 적용 후 | 변화 |
|---|---:|---:|---:|
| 요청 수 | 60,375 | 62,999 | +2,624 |
| TPS | 170.19 | 178.42 | +4.84% |
| 실패율 | 0.02% | 0.00% | 실패 제거 |
| 실패 건수 | 13 | 0 | -13 |
| 평균 응답 시간 | 988.11 ms | 10.23 ms | 약 96.6배 개선 |
| p95 응답 시간 | 4,968.56 ms | 18.85 ms | 약 263.6배 개선 |
| p99 응답 시간 | 5,831.26 ms | 35.90 ms | 약 162.4배 개선 |
| 최대 응답 시간 | 7,000.49 ms | 342.12 ms | 약 20.5배 개선 |

## 5. 해석

Redis 캐시 적용 후 평균 응답 시간이 `988.11 ms`에서 `10.23 ms`로 감소했다.  
p95는 `4,968.56 ms`에서 `18.85 ms`, p99는 `5,831.26 ms`에서 `35.90 ms`로 감소했다.

이 결과는 병목이 단순히 인덱스 유무만의 문제가 아니었음을 보여준다. 인덱스 적용으로 단일 DB 쿼리는 빨라졌지만, 부하 상황에서는 반복되는 인기 콘텐츠 첫 페이지 요청이 DB, 커넥션 풀, 애플리케이션 스레드, 작성자/프로필 조회 비용을 계속 발생시켰다.

캐시 적용 후에는 동일한 인기 콘텐츠 첫 페이지 요청이 Redis에서 처리되면서 DB 접근이 크게 줄었고, 긴 응답 지연 구간이 대부분 사라졌다.

## 6. 실무적 판단

이번 개선은 실무적으로 타당한 방향이다.

- 인기 콘텐츠 목록처럼 반복 조회가 많은 API에 Redis 캐시를 적용했다.
- 실시간 정합성이 아주 강하게 요구되는 데이터가 아니므로 30초 TTL은 수용 가능한 범위다.
- 커서 기반 전체 페이지를 무리하게 캐싱하지 않고 첫 페이지로 범위를 제한했다.
- 생성/수정/삭제 시 캐시를 무효화해 오래된 데이터 노출 시간을 줄였다.

다만 현재 방식은 리뷰 변경 시 `reviewList` 캐시 전체를 비운다. 구현은 단순하지만 콘텐츠 수가 많아지면 캐시 효율이 떨어질 수 있다.

## 7. 다음 개선 방향

1. 콘텐츠 단위 캐시 무효화 검토

현재는 리뷰 생성/수정/삭제 시 `reviewList` 전체 캐시를 제거한다.  
운영 규모가 커지면 변경된 `contentId`에 해당하는 캐시만 제거하는 방향이 더 적합하다.

2. 캐시 hit/miss 모니터링 추가

Redis 캐시 적용 효과를 운영에서 판단하려면 hit ratio를 봐야 한다.  
Micrometer, Redis command 통계, APM 지표를 통해 캐시 적중률을 확인하는 것이 좋다.

3. Cold Cache와 Warm Cache 분리 측정

이번 캐시 적용 후 결과는 Warm Cache 효과가 강하게 반영된 결과로 볼 수 있다.  
운영 판단을 위해서는 서버 재시작 직후 Cold Cache 결과와 캐시가 채워진 Warm Cache 결과를 분리해서 비교해야 한다.

4. DB 부하 지표 함께 확인

응답 시간만으로는 병목 제거 여부를 완전히 판단하기 어렵다.  
부하 테스트 중 PostgreSQL CPU, active connection, slow query, Hikari pool 사용률을 함께 확인해야 한다.

5. 최대 응답 시간 원인 확인

캐시 적용 후에도 전체 HTTP 기준 최대 응답 시간이 `4,217.59 ms`로 기록되었다.  
리뷰 목록조회 API 자체의 최대 응답 시간은 `342.12 ms`로 낮지만, 전체 요청 중 일부 긴 지연이 남아 있는지 별도 확인이 필요하다.

## 8. 결론

Redis 캐시 적용은 리뷰 목록 조회 성능 개선에 효과가 매우 크다.  
특히 인기 콘텐츠 첫 페이지 조회처럼 요청 패턴이 반복되는 API에서는 DB 인덱스만으로 해결되지 않던 고부하 응답 지연을 캐시가 직접적으로 줄여준다.

현재 결과 기준으로는 캐시 적용을 유지하는 것이 맞고, 다음 단계는 콘텐츠 단위 무효화와 캐시 hit/miss 모니터링이다.
