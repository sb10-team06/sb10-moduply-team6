# 프로필 수정 API 동기 S3 업로드 부하테스트 결과

## 테스트 목적

프로필 이미지 수정 로직을 비동기 S3 업로드에서 동기 S3 업로드로 변경한 뒤, 수정 완료 응답 시점에 새 프로필 이미지가 바로 조회되는지 보장하면서도 성능이 운영 수준에서 허용 가능한지 확인한다.

이 문서는 `UserService.updateUser()` 리팩토링 전후 결과를 비교한다.

## 테스트 대상

- API: `PATCH /api/users/{userId}`
- 요청 방식: `multipart/form-data`
- 이미지 파일: `profile-500kb.jpg`
- 이미지 크기: 367,915 bytes
- 인증: `k6-user-*` 테스트 사용자 JWT
- 테스트 스크립트: `k6/user-profile-update.js`

## 리팩토링 전 구조

리팩토링 전에는 `UserService.updateUser()`의 DB 트랜잭션 안에서 S3 업로드가 동기로 수행되었다.

```text
DB 트랜잭션 시작
-> User 조회
-> BinaryContent 저장
-> S3 동기 업로드
-> BinaryContent SUCCESS 변경
-> User profileImg 변경
-> DB 트랜잭션 commit
-> 응답
```

이 구조는 응답 시점에 S3 업로드 완료를 보장하지만, S3 업로드가 끝날 때까지 DB 커넥션을 점유한다.

## 리팩토링 전 30 RPS 테스트

### 부하 조건

```text
10 RPS 1분
20 RPS 2분
30 RPS 3분
0 RPS  30초
```

### 결과

| 항목 | 값 |
|---|---:|
| 총 요청 수 | 7,349 |
| 초당 요청 수 | 17.83 |
| 평균 응답 시간 | 169.12 ms |
| p95 응답 시간 | 511.58 ms |
| p99 응답 시간 | 1,342.69 ms |
| 최대 응답 시간 | 2,619.21 ms |
| 성공 트랜잭션 수 | 7,349 |
| TPS | 17.83 |
| 요청 실패율 | 0.00% |

### 해석

30 RPS까지의 점진적 부하에서는 실패 없이 모든 요청이 처리되었다.

프로필 수정 API는 콘텐츠 조회처럼 고빈도로 호출되는 API가 아니라 이미지 업로드가 포함된 저빈도 쓰기 API다. 따라서 10~30 RPS 구간에서 실패율 0%를 유지한 것은 일반적인 운영 부하 검증 기준에서는 양호한 결과로 볼 수 있다.

다만 p99는 1.34초, 최대 응답 시간은 2.62초까지 증가했다. 동기 S3 업로드가 포함된 API 특성상 일부 요청에서 외부 I/O 지연이 응답 시간에 반영된 것으로 해석된다.

## 리팩토링 전 50 RPS 테스트

### 부하 조건

```text
10 RPS 1분
20 RPS 1분
30 RPS 2분
50 RPS 2분
0 RPS  30초
```

### 관측 결과

50 RPS까지 부하를 올린 테스트에서는 DB 커넥션 풀 고갈이 발생했다.

애플리케이션 로그에서 확인된 주요 에러는 다음과 같다.

```text
HikariPool-1 - Connection is not available, request timed out after 30008ms
(total=30, active=30, idle=0, waiting=129)
```

동일 시점에 트랜잭션 생성 실패도 발생했다.

```text
CannotCreateTransactionException: Could not open JPA EntityManager for transaction
JDBCConnectionException: Unable to acquire JDBC Connection
```

### 해석

50 RPS 구간에서는 HikariCP 커넥션 풀이 모두 사용 중인 상태가 되었다.

| HikariCP 상태 | 의미 |
|---|---|
| total=30 | 최대 커넥션 수 30개 |
| active=30 | 30개 모두 사용 중 |
| idle=0 | 유휴 커넥션 없음 |
| waiting=129 | 커넥션 대기 요청 129개 |

리팩토링 전 구조에서는 S3 업로드 대기 시간이 DB 트랜잭션 시간에 포함된다. 따라서 DB 쿼리 자체가 오래 걸리지 않더라도, S3 업로드 지연으로 인해 DB 커넥션 점유 시간이 길어지고 고부하에서 커넥션 풀이 고갈된다.

## 리팩토링 내용

리팩토링 후에는 S3 업로드를 DB 트랜잭션 밖으로 분리했다.

```text
1. 짧은 DB 트랜잭션으로 사용자 존재 여부 확인
2. DB 트랜잭션 없이 S3 업로드를 동기로 수행
3. 업로드 성공 후 짧은 DB 트랜잭션으로 BinaryContent 저장 및 User profileImg 갱신
4. DB 반영 실패 시 방금 업로드한 S3 파일 보상 삭제
```

이 방식은 응답 시점에 S3 업로드와 DB 반영 완료를 보장한다. 따라서 수정 직후 이미지가 바로 보이는 요구사항은 유지하면서, S3 업로드 시간 동안 DB 커넥션을 점유하지 않도록 개선한다.

## 리팩토링 후 30 RPS 테스트

### 부하 조건

```text
10 RPS 1분
20 RPS 2분
30 RPS 3분
0 RPS  30초
```

### 결과

| 항목 | 값 |
|---|---:|
| 총 요청 수 | 7,301 |
| 초당 요청 수 | 17.96 |
| 평균 응답 시간 | 247.24 ms |
| p95 응답 시간 | 517.02 ms |
| p99 응답 시간 | 5,621.25 ms |
| 최대 응답 시간 | 7,301.18 ms |
| 성공 트랜잭션 수 | 7,301 |
| TPS | 17.96 |
| 요청 실패율 | 0.00% |

### 해석

리팩토링 후 30 RPS 테스트에서도 실패율은 0.00%로 유지되었다. 평균 응답 시간은 리팩토링 전 169.12ms에서 247.24ms로 증가했지만, p95는 511.58ms에서 517.02ms로 거의 동일한 수준이다.

p99와 최대 응답 시간은 리팩토링 전보다 크게 증가했다. 이는 DB 커넥션 풀 병목이라기보다 S3 동기 업로드의 외부 I/O 지연이 일부 요청에 강하게 반영된 결과로 해석된다. 리팩토링의 목적은 응답 시간 자체를 줄이는 것보다, S3 업로드 대기 중 DB 커넥션을 점유하지 않도록 하여 고부하에서 커넥션 풀 고갈을 방지하는 것이다.

## 리팩토링 후 50 RPS 테스트

### 부하 조건

```text
10 RPS 1분
20 RPS 1분
30 RPS 2분
50 RPS 2분
0 RPS  30초
```

### 결과

| 항목 | 값 |
|---|---:|
| 총 요청 수 | 10,035 |
| 초당 요청 수 | 24.30 |
| 평균 응답 시간 | 434.14 ms |
| p95 응답 시간 | 1,940.76 ms |
| p99 응답 시간 | 4,006.79 ms |
| 최대 응답 시간 | 9,623.24 ms |
| 성공 트랜잭션 수 | 10,035 |
| TPS | 24.30 |
| 요청 실패율 | 0.00% |

### 해석

리팩토링 후 50 RPS stress 테스트에서는 요청 실패율이 0.00%로 측정되었다.

응답 시간은 30 RPS 테스트보다 증가했다. p95는 약 1.94초, p99는 약 4.01초, 최대 응답 시간은 약 9.62초까지 증가했다. 이는 S3 업로드를 동기로 유지했기 때문에 외부 I/O 지연이 여전히 API 응답 시간에 반영되기 때문이다.

하지만 리팩토링 전 50 RPS 테스트에서 발생했던 HikariCP 커넥션 풀 고갈과 JDBC 커넥션 타임아웃은 재현되지 않았고, 모든 요청이 성공했다. 따라서 리팩토링은 DB 커넥션 점유 병목을 완화하는 데 효과가 있었다고 판단할 수 있다.

## 결론

리팩토링 전 구조는 30 RPS까지는 안정적으로 처리되었지만, 50 RPS 한계 테스트에서 HikariCP 커넥션 풀 고갈이 발생했다.

리팩토링 후에는 50 RPS stress 테스트에서도 실패율 0.00%를 기록했다. 따라서 S3 업로드를 DB 트랜잭션 밖으로 분리한 개선은 유효하다.

다만 응답 시간 자체는 여전히 S3 동기 업로드 영향을 받는다. 특히 50 RPS 구간에서 p95 1.94초, p99 4.01초, 최대 9.62초가 측정되었으므로, 응답 지연을 추가로 줄이려면 다음 항목을 별도 개선 후보로 검토할 수 있다.

- 이미지 크기 제한 및 압축
- S3 timeout 조정
- S3 업로드 네트워크 지연 모니터링
- presigned URL 기반 클라이언트 직접 업로드 방식

이번 개선의 핵심 성과는 응답 시간을 극적으로 줄인 것이 아니라, 동기 업로드 요구사항을 유지하면서 고부하에서 DB 커넥션 풀이 고갈되는 구조적 병목을 제거한 것이다.
